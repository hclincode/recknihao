# Iter 169 Q2 — No Primary Key and Trino Federation Performance

## Answer

The lack of a primary key itself is almost certainly not causing your slowness — what matters for Trino federation performance is whether the join key and filter columns are **indexed**, not whether a PK specifically exists. Let me walk through what the resources say actually drives performance.

## What Trino actually cares about

**1. Predicate pushdown** — whether Postgres filters rows server-side before returning them to Trino. When Trino sends a query to your log table, it rewrites the WHERE clause and pushes predicates down to Postgres. Whether the table has a PK is irrelevant; what matters is whether Postgres can execute those pushed-down predicates efficiently using an index.

Trino pushes down:
- Equality predicates on any indexed column
- Range predicates on numeric, temporal, UUID, and DATE columns
- IN-lists (generated by dynamic filtering)

Verify with `EXPLAIN (TYPE DISTRIBUTED)` on your slow query. If predicates are embedded inside the `ScanFilterProject` node for the Postgres table, pushdown worked. If you see a separate `Filter` node *above* the scan, Trino is filtering in-memory after pulling the full table — that's the performance killer.

**2. Dynamic filtering** — for the Iceberg-to-Postgres join. Trino derives an IN-list of values from the Iceberg side and pushes it to the Postgres scan at runtime. Use `EXPLAIN ANALYZE` to check `dynamicFilterSplitsProcessed` on the Iceberg side. If it's `0`, dynamic filtering didn't fire.

**3. The join key index** — the actual likely culprit. If you're joining on a column that has no index in Postgres, Postgres will do a full sequential scan regardless of how selective the predicates are. A PK by definition creates an index, so PK tables automatically have an index on the join key. Your log table may simply lack an index on the join column.

## What to check

1. **Run `EXPLAIN (TYPE DISTRIBUTED)`** on the slow join to verify predicate pushdown is happening for the Postgres side.
2. **Check Postgres `pg_stat_activity`** during the query to see the actual SQL Trino sent and whether Postgres is doing a Seq Scan or Index Scan on your log table.
3. **Check if the join column has an index** in Postgres:
   ```sql
   SELECT indexname, indexdef FROM pg_indexes WHERE tablename = 'your_log_table';
   ```
   If the join column isn't indexed, add one: `CREATE INDEX CONCURRENTLY ON your_log_table (join_column_id);`

## The bottom line

For a "log-style table that just accumulates rows," the most common culprit for slow federation joins is:
- No index on the join key (compensate with `CREATE INDEX`)
- Table has grown large enough that even indexed lookups are slow (consider pre-ingesting into Iceberg for repeated analytical use)

The primary key absence is a red herring — it's the absence of a **useful index** that matters for Trino's pushdown performance.

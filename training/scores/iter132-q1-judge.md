## Technical claims verified
- **DELETE alone leaves old Parquet files on disk** — VERIFIED. Iceberg v2 spec: DELETE writes immutable position/equality delete files; original data files are not rewritten and stay referenced by the prior snapshot for time travel. (iceberg.apache.org/spec, /docs/latest/maintenance/)
- **rewrite_data_files with WHERE clause is correct for selectively compacting away delete markers** — VERIFIED. The procedure accepts a `where` filter (e.g. `where => "tenant_id = 'acme'"`) and rewrites only files that may contain matching rows; this is the documented way to apply pending deletes into clean data files. (iceberg.apache.org/docs/1.5.1/spark-procedures/)
- **Trino enforces 7-day minimum (iceberg.expire-snapshots.min-retention default 7d)** — VERIFIED. Trino's `expire_snapshots` `retention_threshold` must be ≥ catalog `iceberg.expire-snapshots.min-retention` (default `7d`) or the call fails. (trino.io/docs/current/connector/iceberg.html)
- **Spark `expire_snapshots` has no equivalent hard floor** — VERIFIED at the procedure level. Spark's procedure accepts `older_than` and `retain_last` directly without a catalog-side minimum-retention enforcement; using `current_timestamp() - INTERVAL '0' DAY` is therefore accepted (subject to table-property `min-snapshots-to-keep` / `max-snapshot-age-ms`, which the answer correctly warns about).
- **CALL syntax for Spark `expire_snapshots(table, older_than, retain_last)`** — VERIFIED against official Iceberg 1.5.1 Spark procedures docs. Named parameters are correct.
- **CALL syntax for Spark `remove_orphan_files(table, older_than)`** — VERIFIED. The procedure supports `older_than` (default 3 days for safety; in-progress writes may otherwise be misclassified as orphans). The answer correctly tells the user to pause ingestion before lowering the cutoff to 1 day. (iceberg.apache.org/docs/latest/maintenance/, /docs/1.5.1/spark-procedures/)
- **`history.expire.min-snapshots-to-keep` and `history.expire.max-snapshot-age-ms` can silently override `retain_last`** — VERIFIED in principle. When `min-snapshots-to-keep` is set higher than the user's `retain_last`, the larger value wins, which would defeat a GDPR purge. The pre-flight UNSET/SET dance is a sound defensive practice. (iceberg.apache.org/docs/latest/maintenance/)

## Errors or gaps
- **LOW**: Step 1 narrative says "Iceberg creates a **delete file**" — this is correct for merge-on-read tables (the production default with Iceberg 1.5.2), but Iceberg also supports copy-on-write (`write.delete.mode = 'copy-on-write'`) where DELETE rewrites whole files instead. The answer doesn't acknowledge this fork. For a regulator-facing runbook the omission is minor because the 4-step purge is still correct either way.
- **LOW**: Statement "The **prior snapshot still exists** and still references the original Parquet files without any deletion markers" is true for the data files themselves but slightly misleading — the prior snapshot's manifest does not reference the *new* delete file (because that file was committed in the new snapshot). Wording is defensible but a careful reader could trip on it.
- **LOW**: `remove_orphan_files` default safety floor (3 days) is not explicitly named; the answer correctly tells the reader to pause ingestion before going below it, but stating "default is 3 days; you must lower it explicitly and pause writes" would be slightly stronger.
- **LOW (completeness)**: The answer mentions retention table properties for `expire_snapshots` but doesn't mention `write.delete.mode` / `write.update.mode` / `write.merge.mode` table properties that affect step 1's on-disk behavior.
- **LOW**: No mention of position-delete vs equality-delete distinction, nor of the new V3 deletion vectors — acceptable for a beginner-targeted GDPR runbook, but worth a sentence in resources for completeness.
- **LOW**: No mention of metadata.json files containing prior snapshot references being themselves replaced — after expire_snapshots, the table's metadata.json is rotated; previous metadata.json files are usually retained per `write.metadata.previous-versions-max` and `write.metadata.delete-after-commit.enabled`. For a strict regulator audit, those older `metadata.json` blobs may still reference (by path string) expired snapshots' files, even though the data files are gone. Not a correctness issue but a completeness nit.

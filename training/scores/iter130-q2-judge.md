## Errors or gaps [HIGH/MEDIUM/LOW]
- **[MEDIUM] `rewrite-all => true` not recommended**. For migrating files from an OLD partition spec into a NEW one, the more idiomatic option is `rewrite-all => 'true'`, which forces rewrite regardless of file-size targets. The answer uses `min-input-files => '1'`, which addresses single-file partitions but does not by itself guarantee that already-large old-spec files get rewritten under the new spec (bin-pack may consider them "done" by size). On a 600M-row table where some monthly partitions already contain large Parquet files, those files could remain under the old spec after the rewrite. Recommend `options => map('rewrite-all', 'true', 'target-file-size-bytes', '268435456')`.
- **[MEDIUM] No mention of Trino OPTIMIZE caveat after partition evolution**. The responder's original draft was right to flag it; removing it loses a real safety note for engineers who instinctively reach for `ALTER TABLE ... EXECUTE optimize` in Trino instead of Spark CALL.
- **[LOW] `$files` query may fail in Trino after partition evolution**. Trino issue #26109 documents that querying `$files` can fail with a schema resolution error when partition field names are reused with different field-ids across specs. The progress-monitoring SQL in Step 3 may need to run from Spark instead in some scenarios. Worth a footnote.
- **[LOW] `min-input-files=1` is aggressive**. It will force rewriting every partition with a single file, including newly-written daily partitions that have not yet accumulated data. For a freshly-evolved spec where new daily partitions intentionally start small, this can be wasteful. Default `min-input-files=5` is usually safer; combine with `rewrite-all=true` to ensure cross-spec migration.
- **[LOW] Timing table is optimistic**. "1 TB in 30-90 minutes" assumes a well-sized Spark cluster (typically 50-100+ executor cores). On a modest on-prem cluster (10-20 cores) shared with other workloads, 1 TB realistically takes 2-4 hours. Worth qualifying as cluster-dependent.
- **[LOW] No mention of partition stats refresh / manifest rewrite**. After rewrite_data_files + expire_snapshots, `rewrite_manifests` is often beneficial to consolidate manifest files (especially when many old-spec manifests have been orphaned). Not strictly required, but a complete maintenance recipe would include it.

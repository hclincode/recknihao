## Key finding

The answer correctly explains Iceberg's delete-file mechanism but presents an incomplete GDPR compliance workflow: it recommends DELETE → rewrite_data_files → sign off, omitting `expire_snapshots`, which is the step that actually removes the old Parquet files (containing the customer's bytes) from MinIO. An engineer following this answer would certify GDPR deletion before the bytes are physically removed from storage.

## Resource gap

The multi-tenant analytics resource (`resources/05-multi-tenant-analytics.md`) needs a "GDPR / right-to-erasure" subsection explicitly covering the three-step physical removal sequence for the production stack: (1) `DELETE FROM table WHERE tenant_id = 'acme'` in Trino creates delete files; (2) `CALL spark_catalog.system.rewrite_data_files(table => '...', where => 'tenant_id = ''acme''')` rewrites affected Parquet files without the deleted rows; (3) `CALL spark_catalog.system.expire_snapshots(table => '...', older_than => TIMESTAMP 'YYYY-MM-DD ...')` removes the old Parquet files from MinIO — this is the step that makes the bytes physically gone and completes GDPR compliance. The subsection should also note that in the production Trino 467 + Spark 1.5.2 stack, steps 2 and 3 are Spark procedures (not Trino DDL), and must be run via the Spark job scheduler (Kubernetes CronJob or equivalent). A GDPR audit checklist (DELETE, rewrite, expire, verify COUNT = 0, verify MinIO file list) would make this directly actionable for a compliance sign-off.

## What was verified correct (via WebSearch)
- "Iceberg stores statistics (min/max) per file for each column — queries with `WHERE browser_type = 'Chrome'` can skip files" — confirmed: Iceberg writes lower_bound/upper_bound per data file in manifest entries, enabling file-level pruning on simple-typed columns. (ryft.io, iceberglakehouse.com)
- "Parquet dictionary-encodes low-cardinality values like `plan_name`" — confirmed: Parquet uses dictionary encoding (combined with RLE + bit-packing) by default for low-cardinality columns, falling back to PLAIN after ~1 MB dictionary size. (clickhouse.com, newsletter.swirlai.com)
- "Iceberg's schema evolution means adding a new column is metadata-only — you don't rewrite old files" — confirmed: ADD COLUMN is a metadata-only op; old files return NULL for the new column ID. (iceberg.apache.org/docs/latest/evolution/)
- "json_extract works but forces full scans — no column pruning, no min/max pruning, no dictionary compression on common values" — confirmed: SQL/JSON functions in Trino require parsing the JSON string at scan time, predicate pushdown for inside-JSON predicates is not supported, and the column is read whole as opaque text. (trino.io/docs/current/functions/json.html, posulliv.github.io)
- Trino `[]` subscript on MAP — confirmed valid operator (trino.io/docs/current/functions/map.html), though answer doesn't mention `element_at()` returns NULL where `[]` errors on missing key.

## Errors or gaps
- **MEDIUM — Spark code has a runtime bug.** The `json_schema` defines four keys (plan_name, browser_type, country_code, feature_flags), then the flatten step calls `col("props_parsed.properties")` — but `properties` is not in the schema, so this line raises `AnalysisException: cannot resolve 'props_parsed.properties'`. The fallback MAP column needs a separate `from_json` (with a `MapType(StringType, StringType)`) over the residual keys, or a `map_filter`/`map_except` step. The engineer will copy-paste this and it will fail. Also, `from_json` schema is declared as a JSON string with type names like `"map<string, boolean>"` — that DDL-style works only via `schema_of_json` or `StructType`/`schema_from_ddl`; the API call as written (passing the schema dict via `from_json`) isn't valid PySpark.
- **MEDIUM — Calls MAP access "parsing JSON per row".** Section "Fallback properties (slower, but necessary for rare queries)" says: "Trino reads the full `properties` column and parses JSON per row." A `MAP<VARCHAR, VARCHAR>` is a Parquet-native nested type with its own encoding — there's no JSON parser involved. It IS true that MAP key/value lookups don't get min/max pruning and that Trino reads the full column, but the mechanism is "scan the entire MAP column and apply `element_at` per row," not JSON parsing. Conflating MAP and JSON storage mechanics may mislead the engineer about what's actually on disk.
- **MEDIUM — Doesn't mention `element_at()` vs `[]` semantics.** The engineer will hit "Key not present in map" errors using `properties['debug_mode']` when the key is absent on some rows. The Trino-idiomatic safe pattern is `element_at(properties, 'debug_mode') = 'true'` which returns NULL for missing keys and lets the WHERE clause naturally drop those rows. This is a frequent footgun for newcomers and the answer misses it.
- **LOW — Doesn't address NULL-for-old-rows on promoted columns.** When the engineer promotes `plan_name` to a top-level column post-migration, all pre-promotion files return NULL for that column. Queries like `WHERE plan_name = 'enterprise'` will exclude all historical rows even if the JSON had `plan_name` set. The answer's storage-cost section is right but doesn't flag this backfill consideration.
- **LOW — `BOOLEAN` in MAP value type is debatable.** The example schema uses `feature_flags MAP<VARCHAR, BOOLEAN>` while `properties MAP<VARCHAR, VARCHAR>`. Mixing value types across maps for similar-purpose data isn't wrong but is inconsistent — no explanation of why feature_flags gets BOOLEAN and properties gets VARCHAR. A two-line note about MAP value-type rigidity (cannot mix types within one MAP) would help.
- **LOW — No mention of Iceberg 1.5.2 / Trino 467 specifics for the prod stack.** The prod_info specifies the exact versions; advice is generally correct for them but the answer makes no claim about MoR vs CoW for the events table (which iter121 just flagged as a recurring resource error). For a high-write events table with MAP columns the user may eventually re-write rows; not strictly required for this question.
